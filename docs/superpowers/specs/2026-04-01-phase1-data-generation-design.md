# Phase 1: Data Generation & Source System Setup — Design Specification

**Date:** 2026-04-01
**Project:** Media Stream Analytics (JioHotstar-style IPL Streaming Platform)
**Scope:** Phase 1 only — data sources and synthetic data generation. No Spark/Airflow pipelines.

---

## 1. Overview

Build a synthetic data generation system that populates MySQL, PostgreSQL, Kafka, and flat files with realistic, joinable datasets simulating an Indian OTT streaming platform (JioHotstar-style). The datasets will later feed a Medallion Lakehouse architecture (Bronze/Silver/Gold) in Phase 2.

### Key Design Decisions

- **Approach:** Modular scripts + shared config (Approach B). Each data source has its own generator script, all reading from a shared `master_ids.json`.
- **Theme:** Hybrid — general OTT content catalog + IPL live match streaming simulation in Kafka.
- **Volume:** Medium/demo-ready. Original entity counts with realistic Indian OTT distributions.
- **Kafka mode:** Configurable — `--mode batch` (default, finite) and `--mode live` (continuous).

## 4. Master Entity IDs — `master_ids.json`

Generated by `master_ids.py`. Loaded by all other scripts via `load_master_ids()`.

```json
{
  "user_ids": ["USR-1000001", "USR-1000002", "...", "USR-1005000"],
  "content_ids": ["CNT-000001", "CNT-000002", "...", "CNT-002000"],
  "campaign_ids": ["CAM-00001", "CAM-00002", "...", "CAM-00050"]
}
```

**Important:** Session IDs (`SES-XXXXXXXX`) are NOT stored here. They are generated dynamically at runtime by `kafka_producer.py` and `generate_file_sources.py` to better simulate streaming behavior.

### ID Formats

| Entity | Format | Example | Count |
|--------|--------|---------|-------|
| User | `USR-XXXXXXX` (7-digit, starting at 1000001) | `USR-1000001` | 5,000 |
| Content | `CNT-XXXXXX` (6-digit, zero-padded) | `CNT-000001` | 2,000 |
| Campaign | `CAM-XXXXX` (5-digit, zero-padded) | `CAM-00001` | 50 |
| Session | `SES-XXXXXXXX` (8-char uppercase hex) | `SES-F7E2A1B9` | Dynamic |
| Event | `EVT-XXXXXXXX` (8-char uppercase hex) | `EVT-9F3A21BC` | Dynamic |

## 5. Database Schemas

### 5.1 MySQL — `streaming_users`

#### `users` table

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `user_id` | VARCHAR(20) | PRIMARY KEY | `USR-1000001` format |
| `email` | VARCHAR(255) | | Faker-generated, some with `+alias` patterns |
| `full_name` | VARCHAR(255) | | Mixed casing noise (ALL CAPS, lowercase, Title Case) |
| `signup_date` | DATE | | 2020-01-01 to 2025-12-31 |
| `device_category` | VARCHAR(20) | | `mobile` (60%), `desktop` (25%), `smart_tv` (15%) |
| `user_agent` | TEXT | | Full UA string matching device_category |
| `country` | VARCHAR(50) | | India 70%, US 10%, UK 10%, Other 10% |
| `created_at` | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP | Record creation metadata |

**Indexes:**
- `INDEX idx_users_country (country)`

**Noise:**
- `full_name`: random casing variations
- `user_agent`: mix of abbreviated and full UA strings
- Some emails with `+alias` patterns (e.g., `user+test@gmail.com`)

#### `subscriptions` table

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `subscription_id` | VARCHAR(36) | PRIMARY KEY | UUID |
| `user_id` | VARCHAR(20) | FOREIGN KEY → users | |
| `plan_id` | VARCHAR(20) | | `FREE`, `BASIC`, `PREMIUM`, `VIP` |
| `start_date` | DATE | | After user's signup_date |
| `end_date` | DATE | NULLABLE | NULL for ~40% active subscriptions |
| `cancel_reason` | TEXT | NULLABLE | Free text, multilingual, typos |
| `created_at` | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP | Record creation metadata |

**Indexes:**
- `INDEX idx_subscriptions_user (user_id)`

**Noise:**
- `cancel_reason`: multilingual free text ("too expensive", "bahut mehenga", "switched to Netflix")
- Some users have multiple subscription records (plan upgrades/downgrades)
- ~40% active (NULL end_date), rest cancelled/expired

### 5.2 PostgreSQL — `streaming_content`

#### `content_catalogue` table

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `content_id` | VARCHAR(20) | PRIMARY KEY | `CNT-000001` format |
| `title` | VARCHAR(500) | | Realistic Indian + international titles |
| `content_type` | VARCHAR(30) | | `movie`, `series`, `sports`, `documentary`, `highlight`, `interview` |
| `genre` | VARCHAR(255) | | Comma-separated, inconsistent formatting |
| `runtime_value` | INTEGER | | Numeric value only |
| `runtime_unit` | VARCHAR(10) | | `seconds` (80%) or `minutes` (20%) |
| `release_year` | INTEGER | | 2015–2025 |
| `language` | VARCHAR(50) | | Hindi 40%, English 25%, Tamil 15%, Telugu 10%, Other 10% |
| `created_at` | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP | Record creation metadata |

**Content type distribution** (maps to `content_type` column):
- `movie` (Bollywood/Indian + International): 35% + 15% = 50%
- `series` (Web series): 25%
- `sports` (IPL matches): 10%
- `highlight` (IPL highlights, player interviews): 5%
- `documentary`: 10%

**Noise:**
- `genre`: inconsistent separators (comma, comma+space, pipe)
- `runtime_unit`: 20% of rows store `minutes` instead of `seconds`
- Some titles with leading/trailing whitespace

#### `ratings` table

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `rating_id` | SERIAL | PRIMARY KEY | Auto-increment |
| `user_id` | VARCHAR(20) | | From master IDs (cross-DB, no FK) |
| `content_id` | VARCHAR(20) | FOREIGN KEY → content_catalogue | |
| `rating_value` | DECIMAL(2,1) | | 1.0–5.0, half stars included |
| `rating_source` | VARCHAR(20) | | `app` (50%), `web` (30%), `tv` (20%) |
| `rated_at` | TIMESTAMP | | Some future timestamps (clock skew) |
| `created_at` | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP | Record creation metadata |

**Indexes:**
- `INDEX idx_ratings_user (user_id)`
- `INDEX idx_ratings_content (content_id)`
- `INDEX idx_ratings_time (rated_at)`
- `INDEX idx_ratings_pair (user_id, content_id)` — improves join performance, allows duplicate detection

**Noise:**
- ~5% duplicate `(user_id, content_id)` pairs (user re-rated)
- Rating distribution skewed positive: most 3.5–4.5
- Some `rated_at` in the future (clock skew)

[
  {
    "event_id": "EVT-9F3A21BC",
    "user_id": "USR-1000042",
    "content_id": "CNT-000315",
    "session_id": "SES-A1B2C3D4",
    "device_category": "mobile",

- 50 rows

### 6.4 TXT Reviews — `data_sources/reviews/`

- Filename: `USR-1000042_CNT-000315_2025-03-15.txt`

- All 2,000 content items get `_v1.jpg`
- ~800 randomly also get `_v2.jpg`
- Resolution: 320x180

## 7. Kafka Streaming Producer

### Topic: `media.viewing.live`

Topic is created explicitly by the orchestrator (not auto-created) using `kafka-topics --create`.

### Event Schema

```json
{
  "event_id": "EVT-9F3A21BC",
  "user_id": "USR-1000042",
  "content_id": "CNT-000315",
  "session_id": "SES-F7E2A1B9",
  "device_category": "mobile",
  "event_type": "PLAY",
  "watch_duration_ms": 185000,
  "seek_position_ms": null,
  "event_ts": "2025-03-15T20:31:05.123Z",
  "session_start_ts": "2025-03-15T20:25:00.000Z",
  "referrer": "home_banner"
}
```

### Event Types & Distribution

| Type | % | Notes |
|------|---|-------|
| PLAY | 35% | Includes ~3% misclassified SEEKs |
| PAUSE | 20% | Spikes during ad breaks |
| SEEK | 15% | Uses `seek_position_ms`, `watch_duration_ms` is null |
| BUFFER_START | 10% | Spikes during peak concurrency |
| BUFFER_END | 10% | Follows BUFFER_START |
| COMPLETE | 10% | End of content |

### Referrer Values

`home_banner`, `search`, `recommendation`, `deeplink`, `push_notification`, `social_share`

### IPL Match Phases (Batch Mode — ~7,500 events)

| Phase | Simulated Duration | Events | Behavior |
|-------|-------------------|--------|----------|
| Pre-match | 30 min | ~500 | Gradual ramp-up, mostly PLAY |
| Innings 1 | 90 min | ~2,500 | High engagement, wicket/boundary spikes |
| Mid-innings break | 20 min | ~500 | PAUSE spike, ad break pattern |
| Innings 2 | 90 min | ~3,000 | Peak load, more BUFFER events |
| Post-match | 30 min | ~1,000 | Rapid drop-off, COMPLETE events |

### Noise

- Session IDs reused across days (~10%)
- Clock skew: `event_ts` ±30 seconds from actual time
- SEEK events sometimes misclassified as PLAY (~3%)

### CLI

```bash
python data_generation/kafka_producer.py --mode batch    # Default, ~7500 events, exits
python data_generation/kafka_producer.py --mode live     # Continuous until Ctrl+C
```

## 9. Orchestrator — `run_phase1.py`

### Execution Flow

```
Step 1: Verify Docker is running
Step 2: docker compose up -d (from docker/ directory)
Step 3: Wait for MySQL ready (retry connection, up to 60s timeout)
Step 4: Wait for PostgreSQL ready (retry connection, up to 60s timeout)
Step 5: Create Kafka topic via `docker exec kafka kafka-topics --create ...`
Step 6: Run master_ids.py → generates master_ids.json
Step 7: Run generate_mysql_data.py → populates streaming_users
Step 8: Run generate_postgres_data.py → populates streaming_content
Step 9: Run generate_file_sources.py → creates CSV, JSON, Excel, TXT, images
Step 10: Print summary + Kafka producer instructions
```

### Logging

- Console output with colored step indicators
- Full log written to `logs/phase1_YYYY-MM-DD_HH-MM-SS.log`
- Each step logs start time, completion time, and row counts

### Usage

```bash
python run_phase1.py
```

## 10. Deterministic Seeds

All generators use fixed seeds for reproducible datasets across runs:

```python
import random
from faker import Faker

SEED = 42
random.seed(SEED)
fake = Faker()
Faker.seed(SEED)
```

This ensures identical output on every run — critical for debugging Spark pipelines in Phase 2.

## 11. Validation Script — `validate_phase1.py`

Runs automatically after `run_phase1.py` (or independently). Checks:

### Database Row Counts
| Table | Expected |
|-------|----------|
| `users` | 5,000 |
| `subscriptions` | 6,000–8,000 |
| `content_catalogue` | 2,000 |
| `ratings` | 15,000–20,000 |

### Cross-System Join Verification
```sql
-- Must return > 0
SELECT COUNT(*) FROM ratings r
WHERE r.user_id IN (SELECT user_id FROM streaming_users.users);
```

### File Existence Checks
- `data_sources/csv/content_metadata.csv` exists and has 2,000+ rows
- `data_sources/json/viewing_events_batch.json` exists and has 50,000+ events
- `data_sources/excel/ad_campaigns.xlsx` exists and has 50 rows
- `data_sources/reviews/` has ~500 .txt files
- `data_sources/thumbnails/` has ~2,800 .jpg files

### Kafka Topic Check
- `media.viewing.live` topic exists

Outputs a pass/fail summary table to console.
